## The Context Chain Principle

**The dot-notation hierarchy IS your context.** Every node in the tree — from root story to leaf micro-issue — contributes a layer of context. When you read a node's assembled prompt via `mtix_context`, you receive the full briefing from root to that node: the business goal, the technical approach, the specific scope, and the exact instruction.

This means two things:
1. **Before executing:** Always call `mtix_context` to get your full briefing. Do not work from the title alone.
2. **Before creating/decomposing:** Write descriptions and prompts that add specific, actionable details. The parent provides the "why." Your child provides the "what" — file paths, function names, edge cases, test expectations. An agent picking up your child node should know exactly what to do.

See [CONTEXT_CHAIN.md](CONTEXT_CHAIN.md) for detailed guidance on writing context-rich descriptions.

## Agent Workflow

The standard agent workflow is:

1. **Start session:** `mtix session start` or `mtix_session_start`
2. **Find work:** `mtix ready` or `mtix_ready` to list available nodes
3. **Read context:** `mtix context <id>` or `mtix_context` — **do this before every task**
4. **Claim:** `mtix claim <id> --agent <agent_id>` or `mtix_claim`
5. **Execute:** Follow the assembled prompt from the context chain
6. **Mark done:** `mtix done <id>` or `mtix_done`
7. **Repeat** until no more ready nodes
8. **End session:** `mtix session end` or `mtix_session_end`

**Step 3 is not optional.** The context chain is how you receive your complete briefing. Skipping it means working with incomplete information.

## When Decomposing Tasks

If you need to break a task into subtasks, use `mtix_decompose`. **Decomposition is a two-step process:**

### Step 1: Create the structure
```
mtix decompose <parent_id> --file plan.jsonl
```
Or via MCP: `mtix_decompose`

### Step 2: Populate every child with context
After creating children, **every leaf node MUST have** `description`, `prompt`, and `acceptance` fields populated. A task with only a title is not actionable — agents cannot pick it up.

```
mtix update <child_id> --description "..." --prompt "..." --acceptance "..."
```

**The CLI will warn you** if you create tasks without these fields. Do not ignore the warning.

For each child node, provide:

- **Description:** The scope, constraints, and why this task exists
- **Prompt:** Exact instructions — file paths, function names, API contracts, test cases, edge cases
- **Acceptance:** How to verify the work is correct — testable criteria

Ask yourself: *"If another agent reads only the assembled context from root to this child, can it execute without asking questions?"* If not, add the missing details.
